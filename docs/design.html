WaiterBot

Overall Design
  • Size: 40cm x 30cm x 60cm
  • Speed: 0.1 m/s - 0.2 m/s
  • Weight - 5kg-7kg
  • Materials used
    • Frame - Using PVC pipes
    • Outer Pannels - Plywood or plastic
Hardware Components Used
  1. Micro Controller - ESP32
    • Specifications and Operating Conditions
      • Oscillator Frequency - 40MHz
      • 34 programmable GPIOs
      • SRAM - 512kB
      • Inbuilt Modules - Wifi, Bluetooth
      • Operating voltage - 3V-3.6V
      • Average current required - 80mA
    • Limitations
      • When wifi is enabled, ADC fucntion of some interfaces will not work properly. To overcome this we are using these interfaces to get digital input when wifi is enabled.
    • Why use ESP32?
      • Has an inbuilt wifi module
      • Has required ammount of GPIOs
      • Less power consumed and batteries will last long
      • High performance due to faster clock and large memory
    ESP32
  2. Obstracle Avoidance - HCSR04 module
    • Uses the principle of reflection of sound to detect an obstracle.
    • No of sensors used - 2
    • Specifications and Operating Conditions
      • Operating voltage - 5V
      • Average current required - 15mA
      • Range - 2cm-400cm
      • Effective Angle - 15 degrees
    • Limitations
      • Distance measurement is less accurate when compared to other distance sensor modules(IR sensors)
      • Detection angle is 15 degrees. To overcome this we are using 2 sensor modules to cover the entire range.
    • Why use HCSR04?
      • We use this module to detect obstrucles and not to measure the distance to the obstrucle. Therefore accuracy not much of a concern when it comes to detecting obstrucles.
      • Dust, light and mist does not effect the reading of the sensor
    HCSR04
  3. Line Following - IR sensor module
    • Uses the principle of reflection of IR light.
    • No of sensors used - 7
    • Specifications and Operating Conditions
      • Operating voltage - 5V
      • Average current required - 40mA
    IR sensor
  4. Drive Train
    Gear Motors - JGA25
    • No of motors used - 2 (driven by back wheels)
    • Specifications and Operating Conditions
      • Operating voltage - 6V-24V
      • Rated voltage - 12V
    • Low rpm
    • High torque
    Motors

    Motor Driver - L298N module
    • Used to control the motor and provide necessary voltage and current
    • No of modules used - 1
    • Specifications and Operating Conditions
      • Operating voltage - 5V
      • Number of channels - 2
    Motor Driver
  5. Power
    Batteries - Lithium ion 18650 3.7V
    • Two voltage levels from battery pack used
      • 5V for the sensors and micro controller
      • 12V for the motors
    • Why lithium ion?
      • Less expensive
      • Can provide enough stall current for the motors
    Batteries

    Charger Module
    • For recharging the battery packs
    Charger Module

    Step Up Module
    • To step down from 12V to 5V
    Step Down Module

    Load cell 5kg

    Load cell 5kg

    HX711

    Load cell outputs a voltage of micro volts. This module is used to amplify the voltage.

Control Unit

  • Main hardware used in the control unit is a laptop.
  • Control unit creates a local network and the WaiterBots are connected to it. This is to isolate the WaiterBots from the internet.
  • In case of a power faliure, the WaitBots can serve already placed orders because internet is not required for the operation of the WaiterBots.
  • Web sockets are used for communication between the control unit and WaiterBots.
  • Control unit will act as the web socket server. And at the same time WaiterBots will act as the web socket clients. We can verify the clients using username, password method or using certificates. TLS is used for encrypting the messages passing through the web sockets and TLS provides encrypted pipes which messages can flow.
  • Control unit is also connected to the cloud server.

Server Backend and Database

Overall Design
Technologies Used
  1. Server - Node.js with express.js
    • Node js is an event-driven environment which has a single threaded event loop as well as async handlers. Because of it’s event loop and async handlers, it has less context switching compared to multi threaded servers.
    • Since our backend has no heavy computations, we have selected Node.js as the WaiterBot system's backend technology.
  2. Database - MongoDB
    • MongoDB is a flexible document based database which has great query performance. It has highly availability and scalability. By using MongoDB we can develop fast and can move fast.
  3. Storage - AWS S3
    • To have better scalability and reliability we use AWS S3 storage as our main storage. All the static content will be stored in the S3 bucket and CDN will access the S3 bucket to deliver the content fast to the users.

Mobile Application

Mobile Application

Ordering the food is mainly done via our mobile app. Customers visit the shop and they scan the QR code near the table using their mobile phones. So we need to target all possible mobile operating systems which the customers have. Here we are mainly targeting the popular iOS and Android operating systems. Then we have to develop our mobile app to support both of these platforms. To overcome this issue we used Flutter cross platform mobile application development framework. Therefore we can develop a single codebase and use it to develop the mobile app for both operating systems mentioned above.

Web Application and Desktop Application

Desgin Architecture
Web Application of Owner
  • We used React.js for the development of the web application. React can build fast, scalable web applications due to its DOM rendering architecture. And it also has a larger community support and a vast number of libraries.
  • Redux is used for state management.
  • Axios is used for communication with the backend server.
  • This is the UI of the restaurant owner. New orders, robot status, adding items to the menu can be done via the web application.
Desktop Application of Operator
  • We used React.js for the development of the web application. React can build fast, scalable web applications due to its DOM rendering architecture. And it also has a larger community support and a vast number of libraries.
  • Redux is used for state management.
  • Axios is used for communication with the backend server.
  • Electron.js was used to build the cross-platform desktop application.
  • This is the UI of the operator. Accepting orders, deploying robot and changing item status can be done via the desktop application.
